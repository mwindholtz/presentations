$59.5 Billion is wasted every year[RTI 2002].  That’s a lot of muda!  The U.S. gross domestic product loses 0.6% a year because of software defects.  Currently, over half of all errors are not found until “downstream” […] or during post-sale software use. This occurs even though vendors already spend 80% of development costs on defect removal.

 

The software industry is becoming increasingly interested in removing defects after they are created.  A focus on defect removal sounds like a good idea.  What if the defects are not created in the first place? What would that be worth?   Is it realistic to suggest that software can be built without defects?  Teams using Lean Software Development are demonstrating that it can be done.

 

Lean Software Development

The software industry finds itself in a situation today similar to automobile manufactures in 1970’s.   Defects are sapping resources and angering customers.  Defects and long cycle times are reducing our ability to respond to new business opportunities.  The most heavily marketed solution is to add more bulky process steps and to do more testing at the end.  This approach lengthens cycle times.  Meanwhile the Lean approach is delivering successful projects using processes with unlikely names like: Scrum, DSDM, Crystal Clear, and Extreme Programming (XP).  Currently the most documented and popular of these processes is Extreme Programming.  Lean Software Development reduces defects and cycle times while delivering a steady stream of incremental business value.

 

Feature cycle time for software development can be measured in the number of days needed between feature specification and production delivery.  This is called: Software In Process (SIP) [Beck 02].  A shorter cycle indicates a healthier project.  A Lean project that deploys to production every 2-weeks has a SIP of 10 working days.  Some Lean projects even deploy nightly.

 

Going to production every 2-weeks may raise fears of introducing new defects.  Good.  Let’s deal with that.  In order to achieve short cycle time we must eliminate defects.  Doesn’t that sound pretty straightforward?  Start by expecting zero defect software production.

Expect Zero Defects

If you expect zero defects a fundamental change in thinking occurs.  Does this mean that no single defect will ever occur?  No.  It means that we treat every defect as something truly unacceptable and change the process to prevent that class of defect from occurring again.  This is very different from rhetorically claiming that defects are unacceptable and exhorting everyone to work harder with the same Multi-Phase processes.

Lean Manufacturing production lines allow each assembly worker to stop the line if a defect appears.  Paradoxically, lean production lines almost never stop.   In “Scientifically Managed” production lines, only the manager can stop the line, yet they often crash because defects are not reported creating bigger problems downstream.  Expect zero defects and you realize that the way you schedule, test, program, and release software all needs to change to accomplish your goal.  And since 80% of current software project dollars are spent removing defects the goal of Zero Defects becomes financially justifiable.

 

It has long been recommend that before typing, programmers think about the problem at hand. Traditionally this has taken the form of an Analysis phase.  There are two inefficiencies in the Analysis Phase.  The first is that it is a phase.  A phase implies a substantial period of time.  That then implies a long feedback loop, long cycle time, and large batch sizes.  All of these are bad news when you are thinking Lean.  The second is that there is no definable goal or deliverable to Analysis.  The value of Analysis is not clearly defined.  If the goal is to: Think about the Problem not the Solution when is that thinking guaranteed to be complete?  Often documents are written in special notation and at some point the analysts think they have done enough and stop.  Another aspect of definable goal that is missing is definable quality measurements.  What is the acceptable quality level of an Analysis effort?  How can this be measured?  This is all very troubling because to program correct code we do need to understand the problem first.   And to be lean we need to deliver in small batches in order to maintain Flow and allow Pull.  The answer lies in translating requirements one requirement at a time (to maintain small batch size) into repeatable automated tests.

 

The Extreme Programming process has a practice of Test-First coding [Beck 03] that is fundamental in achieving the goal of Zero Defects.  Test-First coding requires that an automated unit-test be written before the production code.  The test-to-code cycle time is 5-15 minutes long.  A standard unit test framework called xUnit has been implemented in over 58 programming languages (available for free at: www.xprogramming.com/software.htm).

 

Additionally, automated up-front acceptance testing insures that the correct business value is delivered.  Acceptance tests are built directly from the requirements before the production code is written.  All tests are built one at a time immediately before each piece of production code. Moving the tests to the front of the process gains efficiencies in providing a tight discipline and direction to the team.  As of this writing a promising Acceptance testing framework called fit is available as a beta release. [Fit]

Lean and Extreme

Extreme Programming (XP) is a set of best practices that were found to work in real projects.  XP practices were developed pragmatically. The practices were tested and modified until they fit together.  XP is a bottom-up process developed by programmers.   It works well.  However, is difficult to explain to the uninitiated, why it works.   Michael Hill once remarked: “XP works in practice but not in theory”.  So true.

 

Lean Manufacturing also started bottom-up.   In the late1950’s when Taiichi Ohno improved the time to change the manufacturing dies from one day to 3 minutes[WoJo 91].    Later, the term “Lean” was coined by John Krafcik to explain why this new way of building cars worked so well.   The theory of  “Lean Thinking” matured and was applied to both manufacturing and even the design of new products.

 

Lean Thinking[WoJo 96] defines the 5 principles of: Value, Value Stream, Flow, Pull, & Perfection.  The principles have been applied to many successful businesses in fields from textbook publishing (AtomicDog.com) to travel (SouthWest Airlines).   The principles of Lean closely match what the practices that XP deliver.  (Lean principles and XP practices are compared at the end of this article). Yet Lean Thinking gives the XP practitioner added tools to explain XP productivity and integrate XP into the full range of business concerns.

 

It is the model of Lean Software Development that explains why a business should program Lean.  For example, consider the artifacts of the software process.  Consider the expense that goes into creating those artifacts.  Could these artifacts be inventory?  The Lean Thinking identifies inventory with waste.  So it would be best to figure out how to minimize artifacts.  XP provides the techniques needed to reduce inventory and software in process.

Increased Business Value

There are two ways to increase the business value on a software project: lower the cost or increase the value.  Lower costs can be accomplished with offshore development.  Understandably, corporate executives are unhappy with paying high dollar rates for poor quality software built locally.  Instead they reduce labor costs by moving projects offshore.  Offshore programming gets low-cost & talented programmers to produce the same poor quality software as before.  Reducing cost increases payback.

 

An alternative way to increase payback is to increase project value.  This can be realized by savings of time to market, closer understanding of the real customer requirements, and customer goodwill by providing defect free applications.  Lean Software Development provides the business rationale for this approach. 

 

In the 1980’s Japanese Lean Auto Manufacturers established engineering and production facilities in the U.S.  Did they move development off-(Japanese)-shore for lower U.S. labor rates?  No way!  They now build cars in Ohio and Michigan, in order to be closer to the customer, to ship quicker and to understand more deeply.  They moved closer to customers to produce more profitably.

 

The same occurs when companies adopt Lean Software Development.  Offshore may be less expensive, but when software is business critical, locally built Lean Software Development with short cycle times and low defects are the cost effective option.

Two Case Studies

Case Study 1: Spare Parts Logistics Case Study (SPL)

The Spare Parts Logistics (SPL) project was an 18-month project extending a legacy 5 million LOC logistics system.  The purpose of SPL was to compute depot stock levels of spare parts for items worldwide and to minimize backorders.  The customer organization had millions of parts belonging to tens of thousands of units of equipment that were frequently moved among depot locations.  SPL was an extension of the Previous Logistics Project (PLP).  The names have been all been changed for obvious reasons.  The contract was based on Time and Materials but strong emphasis was placed on delivering on time and budget.  Missing the date or budget would threaten additional contracts expected to provide years of future work.

Motivation for Process Change

Lean Software Development was introduced because of the difficulty experienced in the Previous Logistics Project (PLP).   The PLP had difficulty with code integration and poor software quality resulting in a 700% cost overrun, a 200% schedule overrun, and a generally miserable experience of everyone.  These effects were not foreseen because PLP used a phased waterfall process that indicated to upper management that the project had been going very smoothly.  Phases were time-boxed and milestone defined for the 12 month long project.  Milestones were set for the phases of Requirements, Analysis, Design, Detailed Design, Code Construction, In-house Testing, and Customer Testing.  Each Milestone was accomplished successfully.  However, during Customer Testing it was revealed that the code modules would not work together.  While this was a nasty surprise for management, the programmers themselves had long been expressing concerns.  Programmer concerns remained un-addressed by tech-leads eager to push the forward though hard work and extra effort.  Because of defects, a second 12 months were spent under very unpleasant, high-pressure conditions re-working the application in order to pass the Customer Test phase.  Eventually, PLP passed the Customer tests and went into production.  The system crashed repeatedly over the course of 4 weeks costing a million dollars a day in system outage costs and threatening legal action.  Even the Customer tests were not sufficiently comprehensive.  And major defects first appeared in production.

 

How does this occur?  The reward system for the company was geared toward delivering to milestones.  Milestones and Phases were linked to individual compensation.  So it was in everyone’s interest to push the product passed the next Phase.  These are similarities here in how traditional automotive manufacturers produce inventory and Push it onto their customer using various incentives.  A system built around “Push” often results in unpredictable sub-optimal behavior.  In the end PLP was repaired and continued running in production.

 

Management concluded that a different approach was needed.  A Lean Software Development approach based on XP would be used for the next development project.  So while using the same software environment in the same company culture SPL went to production on time and on budget with no major defects.  In fact, it was four days in production before the first minor (level 4) defect was reported.  Undoubtedly, Lean thinking produced a dramatic turn around.

 

SPL Approach

The SPL development team applied a highly controlled and disciplined approach to Lean Software Development.  This approach is summarized by two imperatives: Working code and Conversing People.

Working Code assures that Flow is preserved.  Working code means that all code is tested by automated tests, and all work is integrated with the whole team multiple times a day.  The code worked and it all worked together all the time. This feedback cycle gave project leadership a high degree of certainty on the true progress of the project.  At project completion 2400 tests were written and running successfully.  Working code provided a medium for measuring Flow of features though the team.

Conversing People assures that waste is minimized.  Conversing People means that we recognize that the highest quality of communication is face-to-face[Co 02].  All too often committing lists of issues to paper slows down communication, prevents questions, and adds administrative work.  Excessive reliance on intermediate documentation can easily become a source of process waste.  Documentation lacks a feedback cycle of question and answer.  When documentation is out of date it can spread confusion and false information requiring added work to correct.  The team including customers and testers work each feature from conception to functioning, clean, tested code.   The Customer provided the Pull of value from the team by specifying features during each Iteration-planning meeting.  Planning took 4-8 hours every two weeks and involved the whole team.

Conversing People also gave the project a high level of transparency.   As new programmers joined in the middle of the project they often remarked how odd it was that we told the customer on a daily basis the state of the project, even when the news was unpleasant.  This transparency quickly helped improve the customer relationship.

 

These imperatives have lead to a project where everyone knew exactly what work had been completed, and unquestionably finished.  The ability to deliver features independently provides a lean software process with the concept of small batch sizes. The code was always working while requirements, designs, and code were continuously improved to increase Flow.

 

SPL Approach Detail

SPL team members followed a set of practices that allowed for Lean Software Development known as Extreme Programming (XP).  There are three groups of practices: Customer practices, Team practices, and Programming practices.

 

The Customer Practices

Iteration Planning

At the beginning of an iteration (lasting 2 – 3 weeks) the whole team met with the requirements team and determined the exit criteria for a successful iteration.  The requirements team determined what needed to be done next providing the Pull of Value. The programming team estimated the effort required.  Then the group calculated what reasonably fit into the allocated time frame.  The meeting goal was to set clear, measurable, realistic, and attainable short-term goals.

On-site Customer

During any software development effort issues occur in layers.  Often questions about requirements are not apparent until the code is being written.  At that point a programmer is forced to either make assumptions or stop work until the correct answer is found.  These lead to defects and re-work if the assumption is wrong, or waiting if work stops.   By seating the domain experts with the team the programmers are given answers as needed, so that programming can progress without work stoppage.  Flow is maintained.  Also a small batch size allows that all questions about the feature currently under development are handled at the same time while fresh in everyone’s mind.

 

Team Practices

Open Workspace

Allowing the programmers, testers, and functional team to sit together in the same workspace enhances communication about project issues.   For example, when development environment problems occurred, they occurred only once.  The first programmer to encounter the problem immediately warned the rest of the team.  This communication required no training.  It was simple.  The first affected programmer let out a spontaneous cry: Oh No the database is down again!  The other programmers overheard and shifted to tasks that did not require the Database. Sometimes another programmer in earshot knew the solution.  At minimum we did not have individuals re-discovering the same problem for themselves.  Problem rediscovery is avoidable waste.

Pair-Programming

There is more to quality programming than typing speed.  Programming is not laying bricks!  Mistakes are expensive.  It is cheaper to review code while it is typed-in.  Also, Pair Programming has helped the team share insights about the design and the tools. Yet another benefit was that no one person became an information bottleneck.  Each task was worked on by at least two people.  If someone was ill at home, the other person picked up the task and continued.  Information bottlenecks were prevented.  Flow was preserved.

Collective Code Ownership

Any pair can extend or fix any piece of code.  On the other hand, under a system of Individual code ownership, individual programmers become production bottlenecks.  Collective Code Ownership only works when unit testing is in place.  The tests guarantee the high code quality.

Coding Standard

The team has striven to write standard code.  The goal is that all code in the system looks similar.  The standard was partially written, partially in the team culture.  They talked freely about how to handle new challenges, and how to code them clearly.  The standard grew out of experience and was updated as new needs appeared.

Sustainable Pace

Some software development projects oscillate between phases of boredom and panic.  This becomes hard to manage because the productivity data swings wildly.  And scheduling of Flow becomes uneven. Sustainable Pace means that the team should work the same hours per week every week.  This provides management with predictable outcomes, and allows the team to be rested and alert while on the job.  Tired programmers write defective code.

 

Programming Practices

Simple Design

Over engineering can easily sidetrack a project by spending too much time on issues not directly related to customer needs.  Simple Design helps a team focus on the simplest thing that fulfills the requirements.  One of the SPL programmers commented that this was the cleanest code he had ever worked on.  The benefit of simpler code is that it is easier to extend and maintain.  In other words, programmers can code faster with simpler designs.  Simpler code is more modular.  One indication of overly complicated design is that it is hard to unit test.  Occasionally in SPL a code change caused multiple unit tests to break. This indicated that the tests were not focused enough and that the design was not simple enough.  Simple design is aided though continuous design improvement (Refactoring).   Simple Design allows for smaller incremental improvements and allows for an ongoing Flow of features.

Continuous Integration

A task is not done until the code is integrated into the single code stream shared by the team.  However, code can be integrated before the entire task is done.  The team could integrate at any point when all the unit tests pass.  In C++ we were able to integrate every 4-6 hours of work.  In Java we would integrate every 20 minutes to 2 hours.  As the length of time between integrations gets longer, the pain in integrating becomes disproportionately greater.  This encourages the programmers to integrate frequently.  Integration occurred at most on a daily basis.  Continuous integration works best in smaller 2-hours cycles.  Our daily integrations caused some work to go slower than it should have because this cycle was too long.

Test First

SPL developed over 2400 tests.  The majority of those were unit tests.  With Test-First programming the test is written before the production code.  We knew we were finished when the tests passed.  This gave clear exit criteria for tasks.  The growing suite of tests provided re-assurance as more features were added.  The team did a great job in testing.  Although they struggled with an underpowered programming environment where compile cycle times were a 20 minutes long.  Such long cycle times forced the unit tests to grow bulkier than otherwise necessary.  The Java part of SPL demonstrated how small and focused the tests could be.  The Java team also showed how cycle times could occur in less than 1 minute.

Refactoring

Left alone software systems tend to slip toward disorder.  As features are added often they extend the original design in unforeseen ways, leading to ever more ugly and brittle code.  Refactoring is the science of improving the design of existing code.  Improving a design allows further new features to be added quickly.  Refactoring in the C++ parts of SPL was minimally applied.  For the Java portion of SPL Refactoring was applied more aggressively and succeeded in exposing both dependency problems and coding errors.  The best way to build flexible code is to flex it.  Refactoring helped revealed design weaknesses early and so the team could fix them.  Refactoring on the C++ part of the project was greatly hampered by the long 20 min. cycle time and the difficulty in creating, and deleting classes in a difficult to use, heavily customized source control system.

 

We noticed that the practices reinforced each other.  For example, if programmers are asked to write test-first code alone, they usually did not write enough tests.  When not enough tests are written continuous Integration becomes more difficult because we didn’t know what was correct and what was broken.  Without Constant Integration, Simple Design is difficult because long integration cycles encourage big elaborate designs.  And so on…

 

Two XP practices were not used for the SPL project.  They were: Small Releases, and Metaphor. 

Small Releases is the practice of releasing to production in small pieces so that real-life production feed back is obtained.  The SPL project contract was for a one time big release after 18 months.  When the contract was written the customer was not aware of incremental approaches to project delivery.

Metaphor attempts to provide conceptual integrity with a unifying mental picture of the application.  By the time a metaphor was suggested for the project, the customer and programmers had already become attached to the technical vocabulary used to describe the system parts.  So they were not interested in switching the names they were already happy using.

Project #1 Highlights

On Time and On Budget

To be precise: the 18-month SPL project was 2-weeks late, and 108% of budget.  The previous project using a plan driven phased-waterfall process with the same organization and technology was 200% of schedule and 800% of budget.  Management and the customer were very pleased with the SPL project.

 

Kaizen or the final sprint

For the first 14 months, the team and management were open to the concept of continuous improvement (Kaizen).  On-going improvements were made both to the process and to the software design.  The team held retrospectives every two-weeks in order to reflect upon and improve team performance.   The suppleness of software provided us with an added tool.  When the team noticed that a repeated task took a long time, they would consider changes to the design as well as to the process.  The ability to quickly change the design to support the work is less available in manufacturing.  But certainly is valuable to apply to Lean Software Development.    Nevertheless, during the last 4 months of the project, the process and design seemed to solidify.    Suggested improvements were fewer.  Management received suggestions for improvement ever more skeptically.  During the final sprint to the end improvement slowed and the focus turned back to working harder.

 

Recovery Team

Half way though the 18-month project, we had enough historical performance data to show that we would miss the deliverable date by two months.  We determined the slip 9 months before the due date.  This early warning provided enough time to react.  Yet at the time upper management was more concerned with a different project that was in crisis.  Three months went by.  Again, we recalculated and presented the data to management.  We still had 6 months left to correct a 2-month shortfall.  The result was the creation of a separate recovery team to separately implement 4 of the remaining features.  The fact that we had deferred design on these features allowed us to shift the design to a new technology for the recovery team.   The recovery team is described in Case Study #2 below.

 

Pulling Value

Early on in the project, the customer team said that a big architectural change would be needed to support a “big dreaded feature”.  During an early iteration planning meeting we talked about the details of the architectural change and the feature involved.   It was truly a big change and the programmers were concerned about how this would affect the schedule.  There was some concern whether it could be accomplished with the current technology.  The conversation was free flowing and wide ranging.  To focus the conversation we asked the customer if he wanted this implemented within the next two weeks.  There was a quite pause.  Then the customer switched gears and suggested that since we needed something right now, we should work on a different feature.  We then defined work for the next two weeks.   Every few iterations, the big dreaded feature would re-emerge.  Each time, we asked the customer to talk in terms of the next two weeks and the big dreaded feature was delayed for a more pressing feature.   Some of the programmers were worried about this approach.  They asked: “if this is so big, won’t we be in trouble if we don’t start to deal with it now?”  But the team focused on providing an even Flow of Value as Pulled by the customer during iteration planning.  Eventually, the customer stopped mentioning the big dreaded feature.   Months later, the team was within 2 iterations of project completion and one of the programmers asked: “what about the big dreaded feature?”  The customer waved him away.  “Well, we couldn’t ever really figure out what that was supposed to do.  So, we will go to production without it.”  This was a huge triumph for letting the customer Pull value while managing the Flow.  It may appear to be pure luck that the feature was not needed, something else is at work.  When the customer Pulls value, non-valuable features naturally fall to the end of the to-do list, as did the big dreaded feature.

 

Feedback and Customer Relationship

The difficult project preceding SPL poisoned the relationship between the company and the customer.  Spending 800% of budget made the customer suspicious.  SPL was able to deliver small bits of working code to the customer at regular 2-week intervals.  Within two months the customer had warmed to the team.  After 8 months the main customer representative requested office space near the team so he did not need to work from a separate building.  The customer representative liked to see that when he specified a feature, it appeared in code a few days later.   At the beginning of the project the customer was detached and suspicious.  And by delivering short feedback loops and allowing him to Pull his own defined Value from the team the relationship improved.

 

Managing Flow

In the early stage of the project, the team had been struggling with test-first programming but managed to work their way up to 178 unit tests.  To encourage them along we wrote the total number of tests on a large whiteboard at the entrance to the shared workspace.   I was gone for 3 days helping out a different client.  When I got back I noticed the number of tests was still stuck at 178!  After three days?  I was shocked.  Had they stopped programming code using the test-first technique?   No, they had stopped programming entirely.  While I was gone the whole team went to meetings with the architect.   Flow had stopped and an architecture phase was inserted.  Architecture is an important element of software development.  But it too can and should be handled in the Flow of delivering business value.  I worked with the team to put practices in place so that future architecture was focused on the Flow of Value.  And they stopped having protracted, unfocused meetings

 

Waste from Rework

One thing clear in retrospect was the waste caused by a lag of weeks between the code being created and the automated customer Acceptance tests being written.  There was a lag of about 2 weeks between the programmers finishing their tasks and the Acceptance test completion.  Inevitably, we would find that even though the unit tests validated all the functionally on the small scale, a number of defects would occur when everything was put together.  In order to correct the defects found in Acceptance testing, the team members had to re-visit the requirements.  Acceptance testing at the end caused rework.  The graph shows the number of tests completed per day.  Acceptance tests are plotted in red and use the scale on the right (0-400).  Unit tests are plotted in dark blue and use the scale on the left (0-2400).  As you can see Acceptance tests were started on day 53 of the project whereas unit testing was applied from the first day of coding.   Acceptance tests should be written immediately after a feature is requested and before the programmers begin work on the feature.  In teams since the SPL project, I have seen a dramatic reduction in wasteful re-work when Acceptance tests are written first. A standard tool for Acceptance testing before code is constructed is the fit framework [Fit]

 

Bottle Necks and small batches

Database Administrator (DBA) time was shared among a number of projects including SPL.  This made it a typical bottleneck resource.  A DBA was given project tasks based when a project expected to experience a “work stoppage”.   A tech lead from another traditional projects once asked: “When will SPL be blocked waiting for the DBA?”  Unfortunately for DBA scheduling purposes, SPL rarely blocked.  Since SPL programmers had small atomic tasks to work on, so they could quickly shift to another small task when a needed resource was unavailable.  This was unfortunate because we did not have the threat of “work stoppage” so it was harder to argue for timely DBA attention.

Case Study 2: Spare Parts Logistics – Forecasting (SPL-F) Team

With 6 months to go in an 18-month schedule we calculated that the SPL project (see above) would be 2 months late.  This was based on historical productivity data generated by the SPL team and on release planning estimates for the C++ based programming team.  We determined that the last 4 features having to do with supply forecasting could be implemented separately.  And decided to use a technology new for the company.  This added risk was justified because programmers in the new technology (Java) would be quicker to find. The SPL-Forecasting (SPL-F) team was the plan to recover the schedule.

 

The manager was very happy with the high productivity and team spirit of the main SPL C++ team.

And all agreed that the main team should not be disturbed.   A separate team would be started from scratch.  This would help us avoid the perils of Brooks law:  Adding people to a late project will make it later. We were adding a separate team to separate features of a late project.  This was possible because our process handled one feature at a time.  The features remained independent enough to break off and assign to this new team.

 

By the time programmers were available we had only 2 months to deliver the application.  They were new to the company, and new to the process, but they had experience with Java and the development environment.

 

We started writing tests, and began to define the code integration cycle.  The Customer team consisted of two in-house domain experts, a test coordinator, and a customer representative.   The SPL-F Team delivered the needed features and made it possible for the SPL to deliver on time and on budget.

Project 2 Highlights

Changing (Increasing) Requirements

It was originally assumed that the requirements were well understood.  The 4 features were a re-implementation of an existing mainframe application.  The goal was to move it to Unix.  The team held an exploration meeting that gave a basic scope to the project.  Next we planned iteration #1.  The goal was to plan work for one week.  The programmers were a bit confused because so much was unclear at the same time.  The team had never used a Lean process, had never programmed in test-first style, and had never integrated software in progress every hour.  They were quite suspicious of the value of Refactoring.  We were nervous since the time frame was short.  They worried that one simple step at a time would not lead to completion in 2 months.  I worried that they may not learn the process in time for productivity to improve.  The time it takes before the productivity boost begins is not related to programmer skill.  It is related to programmer acceptance.  Until the tests are being written with care, few programmers believe that Refactoring can be effective.  The effectiveness comes not only from simplifying the code, but also in a confidence that complex features can emerge from simple features.

 

The original list of 4 features grew to 9.  The original 4 were broken into smaller features and additional features were added to handle issues of concurrency and data-clean-up that did not exist on the mainframe.   Requirements that appear well defined at the beginning often need to be adjusted when the details of implementation are considered.  Lean agility is a good answer to this phenomenon.  But while the list of features grew, the velocity of the team did too.

 

Rapid feedback, Rapid learning

The timeline for the SPL-F team was very compressed yet the team was organized and delivered in 2-months.  We found that a one-week iteration schedule was very powerful for learning and feedback.  Inside of this we also held a brief meeting every morning for 15 minutes to review the previous day’s progress and coordinated current tasks.  The learning and course correction was very beneficial.

 

Wrap up

Lean Principles and XP Practices

In the case studies above we have looked at how the XP practices support Lean principles.  Now we can look more closely at how the principles and practices fit together and what is missing.  Most Lean principles can be implemented by Extreme Programming (XP) practices. The Lean principles are: Value, Value Stream, Flow, Pull, & Perfection.  Considered one at a time how are Lean principles supported by XP practices?

 

On an XP project the customer defines Value.  The XP-Customer refers to whoever provides the programmers with requirements.  An XP-Customer can be the end user, a product manager, or a team of domain experts.  The customer defines the value at three levels.  Release Planning and Iteration Planning (collectively known as the Planning Game) and during daily programming as an On-Site Customer.  During Release Planning the customer defines the features needed for the next small release typically (1-3 months of work).  During Iteration Planning the customer defines features for the next iteration (2-3 weeks of work).  The customer defines automated Acceptance Tests as the criteria for feature completion.  Then as an On-Site Customer, the customer answers fine-grain questions as they occur during the programming day.  A strictly enforced protocol insists that the customer must make all business decisions no matter how small.  The customer defines the Value.

 

XP focuses on the programming aspects of software development.  XP does not provide practices that support the principle of Value Stream.  The Value Stream considers the needs of the customer in the broadest sense and examines how those needs are served from multiple sources.  The programming team is one part of that service.  A fuller application of Value Stream could be a helpful addition to projects using XP to construct software.

 

Flow is supported in XP by many of the practices. 

The team sits together in an Open Workspace so that information is quickly shared.   If one programming pair encounters an environmental problem (i.e. a failed global build, or newly added operating system patch) other programming pairs do not repeat the effort of discovering the same problem.  This allows team members to devote their energy to maintaining Flow in more productive areas of the project work.

 

In traditional projects critical-path analysis is often used to find the shortest way to fit dependent tasks together.  In XP as well as Lean Thinking dependencies are removed and tasks can be done in parallel.  Pair Programming lessens dependency on the knowledge of a particular individual.  If the person responsible for the completion of a task becomes ill, the other programmer continues the work and maintains Flow.   Collective Code Ownership means that anyone can improve any part of the code.  In projects that rely upon individual code ownership work sometimes stops until the designated code owner provides some service.  Sustainable Pace means that the team works a standard schedule from one week to the next.  This creates a more uniform output and allows for more accurate scheduling.  Accurate scheduling then allows management to make wiser interventions to correct longer-term needs.  An example of this was management’s decision to recover the SPL schedule by spinning–off the SPL-F team.

Metaphor while not used in the project above is one way to provide conceptual integrity for a project.  Conceptual integrity is needed for all the team members to understand the inter-working parts of the project.  Simple Design helps a project avoid the ever-present danger of over-engineering.  Over engineering saps productive capacity from a team and redirects it toward seemingly important issues that have no customer relevance.  Simple Design keeps a team focused on the requirements.

Continuous Integration provides a single stream of code that is continuously updated by the various programming pairs.  This insures that even intra-day changes are shared.  In a traditional process integration occurs every evening or every few weeks often causing broken global builds that disturb the production Flow.  Test First programming insurers that features added to the code do not cause rippling defects in other parts of the code.  When the code is tested before it is integrated, integration becomes quick and secure.   Refactoring or continuous design improvement encourages an ongoing simplification of the work product.  The simplification makes defects based on overcomplicated code less likely.  And makes it easier to add future features.

 

The practices of XP that support Pull are similar to the practices that support Value as discussed above.  At the same time the customer defines Value the customer also schedules it for production.   Release & Iteration Planning, On-Site Customer, and customer Acceptance Testing each provide a way to Pull the value from the team.

 

XP teams strive for Perfection in their ongoing efforts to improve the design and process.  While not a defined XP practice, mot XP teams use the Iteration Planning meeting to conduct retrospectives.  This allows continuous process and design improvement.  Collective Code Ownership allows and expects that anyone fix code wherever it is encountered.  Open Workspace allows team members to share insights with one another immediately.  Coding Standard brings uniformity to the code. Test First programming assures that all code is tested.  Refactoring removes duplication and decreases variance.  Pair Programming ensures that all code is reviewed for correctness as it is typed.

Summary

The application of Lean Software Development in the form of the XP process was a great success in the cases described.  The customer was happy, the company made a profit, the software was of high quality, and the programmers felt proud of their work.   Lean thinking required some changes in culture, work habits, and workspace layout.  The success of this project proved the wisdom of making those needed changes.
